# SocialPulse: Predicting Social Media Engagement with a Random Forest Regression Model

*An educational project in applied machine learning and full-stack data-science engineering.*

---

## Abstract

Social media has become a primary channel through which individuals and
organisations communicate with large audiences, yet the factors that make one
post succeed while another goes unnoticed remain difficult to reason about
intuitively. In this project we study whether the engagement rate of a social
media post can be estimated in advance from a small set of attributes that an
author knows before publishing, such as the platform, the format of the post,
the topic, the time of day, and the size of the account. We build a supervised
regression model on a labelled dataset of posts, wrap it in a reproducible
scikit-learn pipeline, and expose it through a small web application so that the
prediction can be explored interactively. Our aim is not to claim a
state-of-the-art result but to work through the complete lifecycle of a
data-science problem, from data preparation and exploratory analysis to
modelling, evaluation, and deployment, in a form that is clear enough to study
and extend. On a held-out test set the model explains roughly three quarters of
the variance in engagement rate, which is a reasonable outcome for a task with
this much inherent noise and serves well as a teaching example.

---

## 1. Introduction

Engagement rate, usually expressed as the proportion of an audience that
interacts with a post through likes, comments, or shares, is one of the most
common ways to measure how well a piece of content performs. Content creators
often rely on experience and intuition when deciding what to post and when to
post it. The motivating question behind this project is a simple one: given only
the information available before a post goes live, how much of its eventual
engagement can we predict?

We treat this as a supervised regression problem. Each post is described by a
fixed set of features, and the quantity we wish to predict is a continuous
engagement rate. The project is deliberately built end to end so that every
stage of the workflow is visible and can be inspected:

1. A dataset of posts with known engagement rates is prepared and cleaned.
2. Exploratory data analysis is used to understand the relationships in the data.
3. A model is trained and evaluated on data it has never seen.
4. The trained model is served through a web interface and a small JSON API.

The remainder of this document follows that structure. It is written to be read
as much as run, so that a student can follow the reasoning behind each choice
rather than only copying commands.

## 2. Data

### 2.1 Description

The dataset used here is a synthetic collection of roughly three thousand posts
stored in `data/social_media_data.csv`. Each row represents a single post and
records both the attributes known at publishing time and the engagement rate the
post ultimately achieved. The data is generated by `generate_dataset.py`, which
builds a realistic but noisy relationship between the features and the target so
that the model has genuine signal to learn while still facing the uncertainty
that characterises real-world content.

We use synthetic data for a practical reason: it lets the project be shared and
reproduced freely, without the privacy and licensing concerns that accompany
real social media data. The trade-off is that results here should be read as a
demonstration of method rather than as findings about any real platform. The
same code can be pointed at a real dataset with the same column names, as
described in Section 7.

### 2.2 Features and target

Each post is described by ten input features, divided into categorical and
numeric groups, and one continuous target.

**Categorical features:** `platform`, `post_type`, `topic`, `day_of_week`,
`sentiment`.

**Numeric features:** `is_weekend`, `post_hour`, `followers`, `hashtags_count`,
`caption_length`.

**Target:** `engagement_rate`, expressed as a percentage.

The categorical variables capture the qualitative context of a post, while the
numeric variables capture its scale and timing. All of these are quantities an
author can know or choose before publishing, which is what makes the prediction
task meaningful in practice.

---

## 3. Method

### 3.1 Preprocessing

Because the features mix categorical and numeric types, they cannot be fed to a
model directly. Categorical variables are transformed with one-hot encoding so
that each category becomes its own indicator, and numeric variables are
standardised so that features measured on very different scales, such as follower
counts and caption lengths, contribute comparably. These transformations are
combined in a `ColumnTransformer` so that each column is handled appropriately.

### 3.2 Model

For the estimator we use a random forest regressor
(`RandomForestRegressor`). A random forest is an ensemble of decision trees, each
trained on a resampled view of the data, whose predictions are averaged. This
choice is well suited to a teaching project: it handles non-linear relationships
and interactions between features without manual feature engineering, it is
robust to the noise present in the data, and it exposes feature importances that
help explain which attributes drive the prediction.

Crucially, the preprocessing steps and the regressor are assembled into a single
scikit-learn `Pipeline`. Keeping the whole transformation-and-model sequence in
one object means that raw feature values can be passed straight to the pipeline
at prediction time, and that the same preprocessing learned during training is
applied consistently afterwards. This avoids a common source of error in
data-science projects, where preprocessing is accidentally applied differently at
training and prediction time.

### 3.3 Post-processing

In addition to the numeric prediction, the estimated engagement rate is grouped
into three interpretable bands, Low, Medium, and High. This mirrors how a
practitioner would actually use such a number: the exact predicted value is
useful, but a coarse category is often easier to act on.

### 3.4 Evaluation protocol

The data is split into a training portion and a held-out test portion, with
twenty percent reserved for testing. The model is trained only on the training
portion and evaluated on the test portion, which it never sees during training.
We report three standard regression metrics: the coefficient of determination
(R squared), the root mean squared error (RMSE), and the mean absolute error
(MAE). Together these describe both how much of the variation the model captures
and how large its errors tend to be in the original units.

---

## 4. Results

On the held-out test set the model typically achieves an R squared of
approximately 0.76, meaning it accounts for about three quarters of the variance
in engagement rate. Given the deliberate noise in the data, this indicates that
the chosen features carry real predictive information without the model simply
memorising the training set. The exact metrics computed during training are
saved to `models/metrics.json` and can be viewed on the Performance page of the
application, alongside the feature importances that show which attributes the
model relied on most.

These results should be understood in the spirit of the project. The point is not
the specific number but the fact that a disciplined pipeline, honest train-test
separation, and interpretable evaluation together produce a result that can be
trusted and reasoned about.

## 5. System architecture

The project is implemented as a small full-stack application so that the model
can be explored rather than only executed from a script. The web layer is built
with Flask, the modelling layer with scikit-learn, and the visual analysis with
matplotlib and seaborn.

The application exposes the following pages and endpoints:

| Page | Route | Purpose |
|------|-------|---------|
| Home | `/` | Overview and headline metrics |
| Dashboard | `/dashboard` | Exploratory charts, per-platform tables, and derived insights |
| Predict | `/predict` | A form for entering post attributes and receiving a prediction |
| Performance | `/performance` | Model metrics and feature importances |
| API | `/api/predict` | A JSON endpoint for programmatic predictions |

The source is organised so that each responsibility lives in one place:

```
Social Media Engagement Analysis/
├── app.py                     # Flask application and routes
├── run.py                     # Bootstrap: build data and model, then serve
├── config.py                  # Central configuration (paths, features, constants)
├── generate_dataset.py        # Creates the synthetic sample dataset
├── requirements.txt
├── README.md
├── test_app.py                # Smoke tests (standard-library unittest)
│
├── data/
│   └── social_media_data.csv  # Sample dataset
│
├── models/
│   ├── train_model.py         # Trains and saves the pipeline
│   ├── engagement_model.pkl   # Saved pipeline (generated)
│   └── metrics.json           # Saved evaluation metrics (generated)
│
├── utils/
│   ├── data_loader.py         # Load and clean the dataset
│   ├── eda.py                 # Charts, aggregated tables, and insights
│   └── predictor.py           # Load the model and make validated predictions
│
├── static/                    # CSS, JavaScript, and generated charts
└── templates/                 # HTML templates for each page
```

## 6. Reproducing this work

The project is intended to be run and modified. The steps below reproduce the
full workflow on a machine with Python 3.9 or newer.

### 6.1 Environment

```bash
cd "Social Media Engagement Analysis"
python3 -m venv venv
source venv/bin/activate        # On Windows: venv\Scripts\activate
pip install -r requirements.txt
```

### 6.2 Running the application

The convenience script `run.py` generates the dataset and trains the model
automatically if they do not yet exist, and then starts the development server:

```bash
python run.py
```

The application is then available at `http://127.0.0.1:5000`.

On macOS, port 5000 is often occupied by the AirPlay Receiver. If the server
fails to start, choose another port:

```bash
PORT=5001 python run.py
```

and open `http://127.0.0.1:5001` instead.

### 6.3 Running each stage manually

To follow the pipeline one stage at a time, the steps can be run separately:

```bash
python generate_dataset.py       # 1. create the dataset
python models/train_model.py     # 2. train and save the model
python app.py                    # 3. start the server
```

### 6.4 The JSON API

Predictions can also be requested programmatically:

```bash
curl -X POST http://127.0.0.1:5000/api/predict \
  -H "Content-Type: application/json" \
  -d '{
        "platform": "TikTok",
        "post_type": "Reel",
        "topic": "Fashion",
        "day_of_week": "Saturday",
        "sentiment": "Positive",
        "post_hour": 19,
        "followers": 20000,
        "hashtags_count": 10,
        "caption_length": 100
      }'
```

The response contains the predicted engagement rate, the corresponding band, and
the validated inputs that were used:

```json
{
  "engagement_rate": 6.52,
  "band": "High",
  "inputs": { "...": "echoed and validated inputs" }
}
```

### 6.5 Tests

A lightweight test suite written with the standard-library `unittest` module
exercises the data pipeline, the model prediction, the routes, and the API. It
requires no additional dependencies:

```bash
python -m unittest test_app -v
```

## 7. Using your own data

To apply the same method to a different dataset, replace
`data/social_media_data.csv` with a file that uses the same column names, which
are defined in `config.py` under `CATEGORICAL_FEATURES`, `NUMERIC_FEATURES`, and
`TARGET`, and retrain:

```bash
python models/train_model.py
```

The dashboard and predictions will then reflect the new data and model. All
paths, feature lists, and category vocabularies are kept in `config.py` so that
the project can be adapted without editing the application logic. A small number
of settings can also be controlled through environment variables:

| Variable | Default | Purpose |
|----------|---------|---------|
| `PORT` | `5000` | Server port |
| `FLASK_DEBUG` | `True` | Toggle Flask debug mode |
| `SECRET_KEY` | development value | Flask secret key (set this in production) |

## 8. Limitations and future work

Several limitations follow directly from the educational scope of this project.
First, the dataset is synthetic, so the specific relationships the model learns
reflect the way the data was generated rather than the behaviour of any real
platform; the natural next step is to repeat the study on genuine, ethically
sourced data. Second, the feature set is intentionally small and captures only
attributes known before publishing, which omits factors such as the actual
content of the caption, the accompanying media, and the broader context in which
a post appears. Third, we consider a single model family; a fuller study would
compare several estimators and tune their hyperparameters systematically. Each of
these is a reasonable direction for a student wishing to take the work further.

---

## 9. Conclusion

This project demonstrates, on a manageable scale, how a predictive question about
social media engagement can be answered with a complete and honest data-science
workflow. By preparing data carefully, analysing it before modelling, keeping
preprocessing and prediction inside a single pipeline, and evaluating on data the
model has never seen, we obtain a result that is both reasonable and
interpretable. Just as importantly, the whole system is arranged so that the
reasoning behind each step can be studied and reproduced, which is the primary
purpose of the work.
